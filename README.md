# README для деплоя Chainlit (версия 0.4.2)

Этот проект содержит **автоматизированный скрипт деплоя** для развёртывания **Chainlit** (UI: чат с LLM) с полным стеком: PostgreSQL (pgvector), Ollama (LLM-сервер), History API и Nginx (HTTPS).  

Скрипт позволяет создавать **изолированные версии** проекта, каждая со своими томами, и управлять ими — деплоить, удалять, очищать старые.

## 🚀 Возможности

- **Полный стек**: Chainlit + PostgreSQL/vector + Ollama (с GPU) + History API + Nginx (HTTPS).
- **Автоматическая сборка** образа Chainlit локально (без попыток скачать с Docker Hub).
- **Уникальные тома** для каждой версии (данные PostgreSQL и Ollama изолированы).
- **Общие модели** Ollama (экономия места на диске).
- **Ручное управление секретами** – вы задаёте `CHAINLIT_AUTH_SECRET` и `POSTGRES_PASSWORD` вручную в шаблоне `.env`, скрипт их не генерирует.
- **Статический URL транскрипции** – вы указываете адрес вашего сервиса транскрипции в `.env` (например, `http://host.docker.internal:<ВАШ ПОРТ>/api/v1/transcribe`).
- **Автоматическая настройка ufw** – скрипт открывает порт `<ВАШ ПОРТ>` для локального хоста и Docker-сети (только если ufw активен).
- **Добавление `extra_hosts`** – автоматически прописывает `host.docker.internal` в compose-файл, чтобы контейнер мог обращаться к сервисам на хосте.
- **Освобождение занятого порта** (интерактивно или принудительно через `--force`).
- **Управление версиями**:
  - список (`--list`)
  - удаление конкретной версии (`--remove N`)
  - очистка старых версий с сохранением последних N (`--clean --keep N`)
- **Манифест** (`.deploy_manifest`) с информацией о ресурсах каждой версии.

## ⚙️ Требования

- **Docker** и **Docker Compose** (установлены).
- **Bash** (для скрипта).
- **openssl** (для генерации секретов и создания SSL-сертификатов, если вы их создаёте сами).
- **NVIDIA GPU** (опционально, для Ollama) – если нет, уберите `deploy.resources` из compose.
- **Свободный порт** для HTTPS (по умолчанию 443) или готовность освободить занятый.
- **Предварительно скачанный дистрибутив Chainlit** – все исходные файлы Chainlit (app.py, модули, статика, конфигурация) должны быть размещены в папке-шаблоне `CHAINLIT_TEMPLATE/`. Скрипт копирует этот шаблон в каждую новую версию, поэтому необходимо заранее подготовить рабочую копию Chainlit с необходимыми кастомизациями.

## 🔧 Настройка

1. **Убедитесь, что скрипт исполняемый**:
   ```bash
   chmod +x ~/dev/docker-containers/deploy.sh
   ```

2. **(Опционально) Настройте порт по умолчанию** в скрипте, изменив `DEFAULT_PORT="443"`, если нужно.

3. **Подготовьте шаблон** – скопируйте все файлы вашего проекта Chainlit в папку `CHAINLIT_TEMPLATE/`. Убедитесь, что там есть:
   - `app.py`
   - `modules/` со всеми модулями
   - `.chainlit/` с конфигурацией
   - `public/` с кастомизациями (если есть)
   - `docker-compose.yml`, `Dockerfile`, `nginx.conf` и другие необходимые файлы.

4. **Создайте файл `.env` в папке шаблона** – это самый важный шаг. Скопируйте содержимое из раздела [Переменные окружения (.env)](#переменные-окружения-env), заполните секреты и укажите свой URL транскрипции. Скрипт использует этот файл как основу для новых версий, подставляя только `PROJECT_NAME` и `NGINX_PORT`.

5. **Подготовьте сертификаты** (если используете HTTPS) – поместите `cert.pem` и `key.pem` в папку шаблона `CHAINLIT_TEMPLATE/certs/`. Скрипт скопирует их в `certs/` новой версии.

6. **Добавьте свои иконки и логотип** в папку `public/` шаблона (см. раздел [Мой вклад в разработку](#мой-вклад-в-разработку)).

### Создание SSL-сертификатов

Если у вас нет доверенных сертификатов, вы можете создать **самоподписанные** для тестирования. Для этого выполните команду в каталоге шаблона:

```bash
cd ~/dev/docker-containers/CHAINLIT_TEMPLATE/certs
openssl req -x509 -nodes -days 365 -newkey rsa:2048 \
  -keyout key.pem -out cert.pem \
  -subj "/C=RU/ST=State/L=City/O=Organization/CN=your-domain.com"
```

**Важно**: 
- Замените `your-domain.com` на реальный домен или IP-адрес вашего сервера.
- Для production-среды используйте сертификаты от доверенного ЦС (например, Let's Encrypt).
- Сертификаты должны располагаться именно в папке `certs/` внутри шаблона, чтобы они копировались в каждую новую версию.

Скрипт автоматически скопирует их в новую версию, и Nginx будет использовать их для HTTPS.

### Генерация секретов и создание .env

Скрипт **не генерирует** секреты автоматически – он копирует файл `.env` из шаблона и заменяет в нём только `PROJECT_NAME` и `NGINX_PORT`. Поэтому вы должны подготовить шаблон `.env` со своими секретами.

1. **Сгенерируйте секретные ключи** с помощью команды:
   ```bash
   openssl rand -hex 32
   ```
   Это выдаст строку длиной 64 символа (например, `7a1b973756b3b08c7db06b57e40411993d2a3af802492bae940628e0ca027742`).

2. **Создайте файл `.env`** в папке `CHAINLIT_TEMPLATE/` со следующим содержимым (замените `CHANGE_ME` на сгенерированные строки и укажите свой адрес транскрипции):

   ```env
   # === Автоматически подставляются скриптом ===
   PROJECT_NAME=chainlit-dev-0
   NGINX_PORT=443

   # === Задайте вручную (сгенерируйте через openssl rand -hex 32) ===
   CHAINLIT_AUTH_SECRET=7a1b973756b3b08c7db06b57e40411993d2a3af802492bae940628e0ca027742
   POSTGRES_PASSWORD=strong_password_here

   # === Статические настройки ===
   ASSISTANT_ROLE=Разработчик
   ROLES_FILE=roles.json

   # === URL вашего сервиса транскрипции ===
   AUDIO_TRANSCRIBER_API_URL=http://host.docker.internal:<ВАШ ПОРТ>/api/v1/transcribe
   AUDIO_TRANSCRIBER_BATCH_URL=http://host.docker.internal:<ВАШ ПОРТ>/api/v1/transcribe/batch
   ```

   **Важно:** 
   - `PROJECT_NAME` и `NGINX_PORT` будут перезаписаны скриптом, но для шаблона они могут быть любыми.
   - В секретах используйте сгенерированные значения.
   - URL транскрипции должен указывать на ваш микросервис (см. раздел [Микросервис транскрипции](#микросервис-транскрипции)).
   - Замените `<ВАШ ПОРТ>` на реальный порт, на котором работает ваш сервис транскрипции (например, 8100).
   - *Лучше развернуть на высоком порту - сложнее для ботов.*

3. **Сохраните файл**. При каждом деплое новой версии скрипт скопирует этот `.env` и подставит в него уникальные `PROJECT_NAME` и `NGINX_PORT`.

## 📦 Использование

Все команды выполняются из каталога `~/dev/docker-containers`.

### 🟢 Деплой новой версии

```bash
./deploy.sh --port 443 --build
```

- `--port <порт>` – внешний HTTPS-порт (по умолчанию 443).  
- `--build` – принудительно пересобрать образ Chainlit (рекомендуется при первом деплое или после изменений в коде).  
- `--force` – автоматически освобождать занятый порт без подтверждения.
- `--auto` – полностью автоматический режим (выбор порта по умолчанию, принудительное освобождение).
- `--purge` – удалить все старые версии перед деплоем (полная очистка).

**Пример автоматического деплоя с полной очисткой:**
```bash
./deploy.sh --auto --purge
```

После успешного деплоя:
- Контейнеры запущены, тома созданы.
- Сервис доступен по `https://<ваш-сервер>:<порт>`.
- В папке `CHAINLIT_DEV_<N>` созданы (скопированы) `.env`, `roles.json`, `.deploy_manifest`.

### 📋 Управление версиями

**Посмотреть все развёрнутые версии (с портами и датами):**
```bash
./deploy.sh --list
```

**Удалить конкретную версию (номер N):**
```bash
./deploy.sh --remove N
```
Эта команда останавливает контейнеры, удаляет тома, сеть, образы и папку проекта.

**Очистить старые версии, оставив последние K:**
```bash
./deploy.sh --clean --keep K
```
Например, оставить последние 2 версии:
```bash
./deploy.sh --clean --keep 2
```

### 🧹 Очистка системы

Если необходимо удалить **все следы** проекта (включая тома и папки), выполните:

```bash
# Удалить все папки версий
sudo rm -rf ~/dev/docker-containers/CHAINLIT_DEV_*

# Удалить все тома
docker volume ls -q | grep chainlit-dev | xargs docker volume rm

# Удалить сети
docker network ls -q | grep chainlit-dev | xargs docker network rm

# Удалить образы chainlit-app
docker images -q --filter "reference=chainlit-app" | xargs docker rmi

# Очистить неиспользуемые ресурсы Docker
docker system prune -f
docker volume prune -f
```

> **Примечание:** скрипт `deploy.sh --purge` уже выполняет это автоматически, но если нужно вручную – используйте эти команды.

## 🔐 Переменные окружения (.env)

Скрипт **копирует** шаблон `.env` из `CHAINLIT_TEMPLATE` в новую папку и заменяет в нём `PROJECT_NAME` и `NGINX_PORT` на актуальные значения. Все остальные переменные остаются как в шаблоне – вы задаёте их один раз в шаблоне, и они используются во всех версиях.

**Обязательные переменные, которые вы задаёте вручную:**

| Переменная | Описание | Пример значения |
|------------|----------|-----------------|
| `CHAINLIT_AUTH_SECRET` | Секретный ключ для JWT-аутентификации | `7a1b973756b3b08c7db06b57e40411993d2a3af802492bae940628e0ca027742` |
| `POSTGRES_PASSWORD` | Пароль для базы данных PostgreSQL | `strong_password_here` |
| `AUDIO_TRANSCRIBER_API_URL` | URL эндпоинта транскрипции одного файла | `http://host.docker.internal:<ВАШ ПОРТ>/api/v1/transcribe` |
| `AUDIO_TRANSCRIBER_BATCH_URL` | URL для пакетной транскрипции (если поддерживается) | `http://host.docker.internal:<ВАШ ПОРТ>/api/v1/transcribe/batch` |

**Автоматически подставляемые переменные:**

| Переменная | Значение | Примечание |
|------------|----------|------------|
| `PROJECT_NAME` | `chainlit-dev-<N>` | Уникальное имя проекта (номер версии) |
| `NGINX_PORT` | выбранный порт | Внешний HTTPS-порт |

Остальные переменные (например, `ASSISTANT_ROLE`, `ROLES_FILE`) вы также можете задать в шаблоне.

Все переменные используются в `docker-compose.yml` через подстановку `${...}`.

## 🎙️ Микросервис транскрипции

Для распознавания аудио и видео в чате требуется отдельный микросервис, который принимает аудиофайлы и возвращает текст. В этом проекте мы используем сервис на базе **Faster Whisper**, развёрнутый отдельно (не входит в данный стек).

**Настройка:**

1. Убедитесь, что ваш микросервис транскрипции запущен и доступен.
2. В шаблоне `.env` укажите полный URL эндпоинта для транскрипции одного файла:
   ```env
   AUDIO_TRANSCRIBER_API_URL=http://host.docker.internal:<ВАШ ПОРТ>/api/v1/transcribe
   ```
   (замените порт и путь на ваши)

3. Если сервис поддерживает пакетную обработку, укажите также:
   ```env
   AUDIO_TRANSCRIBER_BATCH_URL=http://host.docker.internal:<ВАШ ПОРТ>/api/v1/transcribe/batch
   ```

**Если у вас нет такого сервиса** – вы можете создать его самостоятельно или обратиться за помощью (см. [Контакты](#контакты)). Скрипт деплоя не включает развёртывание транскрибатора, так как он может быть запущен отдельно или в составе другого стека.

## ⚙️ Как это работает

1. Скрипт определяет номер следующей версии (`CHAINLIT_DEV_<N>`).
2. Копирует шаблон `CHAINLIT_TEMPLATE` в новую папку.
3. Копирует сертификаты (если есть).
4. Запрашивает или принимает порт.
5. Копирует `.env` из шаблона и подставляет в него `PROJECT_NAME` и `NGINX_PORT`.
6. Копирует `roles.json` (если есть) или создаёт дефолтный.
7. Добавляет `extra_hosts` в `docker-compose.yml` (если отсутствуют) для поддержки `host.docker.internal`.
8. Настраивает `ufw` для открытия порта транскрипции (порт, указанный в `.env` как `<ВАШ ПОРТ>`) для локального хоста и Docker-сети (если ufw активен и есть права sudo). По умолчанию скрипт настраивает порт 9090, но вы можете изменить это поведение, отредактировав скрипт.
9. Собирает образ `chainlit-app:chainlit-dev-<N>` (если указан `--build` или образ отсутствует).
10. Запускает `docker compose up -d` (без `image:` – используется локальный образ).
11. При первом запуске (или если модели отсутствуют в общем томе) сервис `ollama-init` автоматически скачивает необходимые модели. Этот сервис выполняется один раз и после завершения останавливается (не перезапускается). Контейнер `ollama-init` **не удаляется**, но после успешного выполнения он находится в состоянии `Exited` и не влияет на работу остальных сервисов.
12. Создаёт манифест `.deploy_manifest` для отслеживания ресурсов.
13. Если указан `--clean` или `--purge` – удаляет старые версии (с принудительным использованием `sudo rm -rf` для папок).

## 🎨 Мой вклад в разработку

Данный проект является результатом **вайбкодинга** — всё опробовано на реальном сервере и работает стабильно. Основная кастомизация направлена на **изменение внешнего вида и поведения** Chainlit без изменения исходного кода фреймворка.

### Кастомизация UI

Вы можете настроить интерфейс через файлы:

- **`.chainlit/config.toml`** — основные настройки Chainlit (логотип, цветовая схема, поведение чата, системные подсказки).
- **`public/custom.css`** — пользовательские стили (переопределение стандартных классов, изменение шрифтов, цветов, расположения элементов).
- **`public/custom.js`** — пользовательские скрипты (добавление функциональности, например, автоматическое раскрытие подсказок, кастомные действия при отправке сообщения).

### Добавление брендинга

Для полной смены визуального стиля поместите в папку `public/` шаблона (и она автоматически скопируется в каждую новую версию) следующие файлы:

| Файл | Размер / формат | Назначение |
|------|-----------------|------------|
| `icon-192.png` | 192×192, PNG | Иконка для веб-приложения (favicon, PWA) |

## ⚠️ Известные ограничения

### Интеграция PostgreSQL (History API)

На данный момент реализована **базовая интеграция с PostgreSQL** для хранения истории чатов:

- ✅ **Создание** новых тредов — работает.
- ✅ **Чтение** тредов — работает.
- ✅ **Удаление** тредов — работает.
- ❌ **Редактирование** существующих тредов (переименование, изменение содержимого) — **не поддерживается**.

Это связано с тем, что текущая реализация History API (на основе Prisma) не предусматривает обновление уже созданных записей. Если вы хотите добавить возможность редактирования тредов, потребуется доработать:
- Модель данных в `prisma/schema.prisma` (добавить поля `updated_at` и разрешить обновление).
- API-эндпоинты в `history-api` (добавить `PUT` или `PATCH` методы).
- Интерфейс Chainlit (добавить кнопку редактирования названия треда).

Буду рад принять пул-реквесты с реализацией этой функции!

## ❗ Возможные проблемы

| Проблема | Решение |
|----------|---------|
| **Ошибка `pull access denied`** | Убедитесь, что в `docker-compose.yml` сервиса `chainlit` отсутствует строка `image:`. Образ должен собираться локально через `build: .`. |
| **Порт 443 занят** | Используйте `--port 8443` или остановите текущий nginx-контейнер (скрипт предложит). |
| **Не стартует база данных** | Проверьте логи: `docker compose -p chainlit-dev-<N> logs db`. Убедитесь, что порт 5432 не занят другим контейнером. |
| **Ollama не загружает моделей** | Проверьте интернет-соединение и логи `ollama-init`. Если модель уже есть, можно закомментировать `ollama-init` в compose. Модели скачиваются автоматически при первом запуске в общий том (например, `chainlit-dev-ollama-models`), поэтому при создании новой версии они не скачиваются заново. |
| **Chainlit не видит транскрибатор** | Проверьте, что сервис транскрипции запущен и доступен. Убедитесь, что в `.env` указан корректный URL (с `host.docker.internal` или IP хоста) и правильный порт `<ВАШ ПОРТ>`. Также проверьте, что `ufw` не блокирует этот порт (скрипт настроит автоматически, но если нет – добавьте вручную). |
| **Ошибка 401 Invalid authentication token** | Очистите куки и localStorage браузера для вашего домена (или откройте страницу в инкогнито). Если вы изменили `CHAINLIT_AUTH_SECRET`, старые токены станут невалидными. |
| **Контейнер не видит хост по `host.docker.internal`** | Убедитесь, что в `docker-compose.yml` для сервиса `chainlit` добавлена секция `extra_hosts: - "host.docker.internal:host-gateway"`. Скрипт делает это автоматически. |
| **Ошибка удаления папок (Permission denied)** | Скрипт использует `sudo rm -rf` для папок, если обычный `rm` не срабатывает. Убедитесь, что у пользователя есть права sudo без пароля (или скрипт запросит пароль). |
| **Порт транскрипции недоступен** | Проверьте настройки ufw. Если скрипт не настроил автоматически, выполните вручную: `sudo ufw allow from 127.0.0.1 to any port <ВАШ ПОРТ> proto tcp` и `sudo ufw allow from 172.17.0.0/16 to any port <ВАШ ПОРТ> proto tcp`. |
